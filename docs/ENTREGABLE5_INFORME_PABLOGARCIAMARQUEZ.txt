1. OBJETIVO
================================================================================

El objetivo de este entregable es diseñar e implementar una solución cloud
completa que integre una API backend contenerizada, una base de datos en Azure,
un registro de imágenes Docker y un pipeline de integración y despliegue continuo.

La aplicación desarrollada es un agente RAG (Retrieval-Augmented Generation)
con FastAPI. El sistema recupera fragmentos de conocimiento almacenados en
Azure SQL Database, los utiliza como contexto y devuelve respuestas trazables
con fuentes (título, origen y puntuación de relevancia).

El resultado final es una API pública desplegada en Azure Container Apps (ACA),
alimentada por imágenes publicadas en Azure Container Registry (ACR) y
automatizada mediante Azure DevOps con un agente self-hosted en Windows.

2. ARQUITECTURA DE LA SOLUCIÓN
================================================================================

El flujo de la arquitectura es el siguiente:

  Desarrollador -> Azure Repos (código + azure-pipelines.yml)
       |
       v
  Azure DevOps Pipeline (agente NAVE-NODRIZA, pool Default)
       |-- Stage Validate: pytest
       |-- Stage BuildAndPush: docker build + push
       |-- Stage Deploy: actualización de Container App
       |-- Stage ValidateDeployment: pruebas HTTP en cloud
       v
  Azure Container Registry (acragent5pgar / rag-agent-api)
       v
  Azure Container Apps (cae-entregable5-rag, ingress externo :8000)
       v
  Azure SQL Database (ragdb / dbo.knowledge_chunks)

Recursos principales en el grupo rg-entregable5-rag (France Central):

  - SQL Server: sql-rag-ent5-pgar
  - Base de datos: ragdb
  - ACR: acragent5pgar
  - Container App: cae-entregable5-rag
  - URL pública: https://cae-entregable5-rag.thankfulmushroom-31f056e1.francecentral.azurecontainerapps.io

4. BASE DE DATOS AZURE SQL
================================================================================

Se provisionó Azure SQL Database ragdb en el servidor sql-rag-ent5-pgar con
autenticación SQL (usuario sqladminrag). La tabla dbo.knowledge_chunks almacena
el conocimiento del agente: id, title, source, content, embedding y metadatos.

Medidas de seguridad aplicadas:
  - Cadena ODBC solo en .env local y en Variable Group secreta del pipeline.
  - Firewall con IP del cliente y regla Allow Azure services.
  - En ACA, la cadena se inyecta mediante secretref, no en texto plano.

Tras ejecutar init_db.py se cargaron 4 fragmentos de conocimiento iniciales,
validados en el Query editor del portal.

7. PIPELINE CI/CD CON AZURE DEVOPS
================================================================================

Por las limitaciones de Azure for Students (sin parallelism en hosted agents),
se utilizó un agente self-hosted Windows (NAVE-NODRIZA) en el pool Default,
con Docker Desktop y Azure CLI disponibles.

Configuración del pipeline:
  - Variable Group vg-entregable5-rag (AZURE_SQL_CONNECTION_STRING como secreto).
  - Service Connection SC-AZURE (compartida entre proyectos DevOps).
  - Stages: Validate, BuildAndPush, Deploy, ValidateDeployment.

El run #20260713.3 completó los cuatro stages en verde en aproximadamente
siete minutos: tests, build y push a ACR, actualización de ACA y validación
HTTP automática de /, /health, /knowledge y /ask.

9. REFLEXIÓN Y BUENAS PRÁCTICAS
================================================================================

RAG frente a una API tradicional:
  Una API REST convencional responde con datos fijos o reglas programadas.
  El agente RAG añade recuperación semántica desde una base de conocimiento,
  lo que permite respuestas contextualizadas y auditables mediante el campo
  sources. En este proyecto Azure SQL cumple el papel de almacén estructurado
  de fragmentos, separando datos de la lógica de la aplicación.

Protección de secretos:
  No se subió .env al repositorio. La cadena ODBC se gestiona como secreto en
  Azure DevOps y como secretref en Container Apps. Las capturas del informe
  ocultan contraseñas cuando aparecen en pantalla.

Lecciones del despliegue:
  - El firewall de Azure SQL debe incluir la IP del cliente y permitir servicios
    de Azure para que ACA pueda conectar.
  - En Azure for Students es necesario un agente self-hosted; vmImage hosted
    no es viable por límites de parallelism.
  - Las Service Connections de DevOps son por proyecto; compartir SC-AZURE entre
    proyectos requiere autorización explícita o creación en cada uno.
